# IJSaP Desk Review Web Tool

A web-based desk review decision tool for the International Journal for Students as Partners (IJSaP). Used by editorial teams when a manuscript is assigned to them in OJS. The tool walks editors through a three-layer decision rubric, collects their assessment, generates a draft email to the author, saves a record to a secure database, and produces a shareable URL so a second co-editor can review the assessment before it is sent.

## Prerequisites

You need three accounts. All should be registered with a journal role email address (not a personal one) for succession and handover purposes.

| Account | What it holds | Credentials kept in |
|---|---|---|
| GitHub | Source code repository (under a journal-owned account or org) | GitHub account credentials |
| Netlify | Static hosting and serverless functions | Netlify dashboard, linked to GitHub repo |
| Supabase | PostgreSQL database (hosted) | Supabase dashboard; service role key stored in Netlify env vars |

## 1. Supabase Setup

1. Create a new Supabase project. **Choose the Canada (Central) region** at project creation, since records will contain submission titles and, in saved email text, author names.
2. Go to the SQL Editor in the Supabase dashboard.
3. Paste the contents of `supabase-schema.sql` and run it. This creates the `desk_reviews` table, enables Row Level Security, and adds a deny-all policy.
4. Copy the **Project URL** from Settings > API (it looks like `https://abcdefg.supabase.co`).
5. Copy the **service role key** (under "Project API keys"). Never use the anon key anywhere in this project.

## 2. Netlify Setup

1. Log in to Netlify and click "Add new site" > "Import an existing project".
2. Connect the GitHub repository.
3. Build settings should auto-detect from `netlify.toml`:
   - Publish directory: `/`
   - Functions directory: `netlify/functions`
4. Go to **Site configuration > Environment variables** and add these three:

| Variable | Value |
|---|---|
| `SUPABASE_URL` | Your Supabase project URL |
| `SUPABASE_SERVICE_KEY` | Your Supabase service role key |
| `IJSAP_SECRET` | A long random string, at least 20 characters, generated by a password generator. This is the sole barrier to API access. Never use a memorable word or phrase. |

5. Deploy the site. Copy the site URL (e.g. `https://your-site.netlify.app`).

## 3. GitHub Keep-Alive Setup

The Supabase free tier pauses projects after extended inactivity. A GitHub Actions workflow sends one request per week to prevent this.

1. In the GitHub repository, go to **Settings > Secrets and variables > Actions**.
2. Add a new **Repository secret** named `IJSAP_SECRET` with the same token value used in Netlify.
3. Add a new **Repository variable** named `SITE_URL` with your Netlify site URL (e.g. `https://your-site.netlify.app`).
4. The workflow at `.github/workflows/keepalive.yml` runs every Sunday at 06:00 UTC and can also be triggered manually from the Actions tab.
5. If the workflow run fails (visible in the Actions tab), it means the API is not responding with HTTP 200. Check that the Netlify deployment is live and the environment variables are correct.

## 4. Post-Deployment Configuration

1. Open `content.js` in the repository.
2. Find the `proposalTemplateURL` constant in the DEPLOYMENT CONSTANTS block near the top.
3. Replace `"SET_AT_DEPLOYMENT"` with the actual URL of the journal's proposal template.
4. Commit and push. Netlify redeploys automatically.

## 5. OJS Email Template Setup

1. In OJS, go to **Settings > Workflow > Emails**.
2. Find the **EDITOR_ASSIGN** template (the email sent when a submission is assigned to an editor).
3. Add the following link to the template body, on its own line:

```
https://[your-site-url]/?submissionID={$submissionId}&title={$submissionTitle}&section={$sectionTitle}
```

Replace `[your-site-url]` with your Netlify site URL.

**Cautions:**

- Verify the template variable names (`{$submissionId}`, `{$submissionTitle}`, `{$sectionTitle}`) against the installed OJS version's template editor. Variable names can vary between OJS releases.
- Test with a dummy submission whose title contains an ampersand (`&`) and quotation marks (`"`). If the title arrives truncated or mangled, use the fallback: a link with `submissionID` only, and have editors paste the title manually.

## 6. Distributing the Access Token

Distribute the `IJSAP_SECRET` value to editors as their access token. Share it through a trusted channel (e.g. a direct message in a secure platform), not by email. Every editor uses the same token.

## 7. Content Update Workflow

Records store `CONTENT_VERSION` and `RUBRIC_VERSION`, so past records always show which text was in force. Three update paths:

### Editing existing text

1. Change the text in the canonical markdown content document (`IJSAP_WebTool_Content_v1.md`).
2. Mirror the change in `content.js`.
3. Bump `CONTENT_VERSION` in both files.
4. Commit and push (Netlify redeploys automatically).

### Adding new blocks

1. Add a new ID and text in the markdown document.
2. Add a matching entry in `content.js`.
3. If the new block requires new behaviour, update `index.html`.

### Version tracking

- Never rename or delete an ID once records reference it.
- The markdown document is the single source of truth: `content.js` must be regenerable from it at any time.

## 8. Updating the Rubric Version

When the Editorial Board adopts a revised rubric:

1. Update `RUBRIC_VERSION` in `content.js` to reflect the new version.
2. Update the corresponding constant in the markdown content document.
3. Commit and push.

## 9. History View

Access the history view at `https://[your-site-url]/history.html`. It requires the same access token. The history view shows summary statistics, filterable records, and a CSV export.

## 10. Backup Routine

Quarterly, open the history view with no filters and click "Export to CSV". Store the CSV file in the journal's Google Drive. Supabase free-tier backups are short-lived, so this bounds the worst case to one quarter of records.

## 11. Rotating the Shared Secret

1. Generate a new token (at least 20 random characters).
2. Update the `IJSAP_SECRET` environment variable in **Netlify** (Site configuration > Environment variables).
3. Update the `IJSAP_SECRET` secret in **GitHub** (Settings > Secrets and variables > Actions).
4. Inform editors of the new token through a trusted channel (not email).
5. Editors will be prompted to re-enter the token on next use.

## 12. Portability Note

The frontend (`index.html`, `history.html`, `content.js`, `style.css`) communicates with the backend through relative `/api/*` paths using a simple HTTP contract (documented in the build prompt). If institutional infrastructure becomes available (e.g. McMaster or MulPress hosting), only the backend implementation needs to change: any static file host plus any HTTPS endpoint set implementing the API contract plus any relational database is an acceptable substitute. The frontend files are implementation-independent.
